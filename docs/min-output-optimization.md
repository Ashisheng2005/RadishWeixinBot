# 最小模型输出实现优秀效果

## 目标
- 在保证正确率和任务完成率的前提下，降低输入/输出 token。
- 减少无效工具调用轮次，提高首轮命中率。
- 通过指标持续迭代，而不是一次性“调参碰运气”。

## 优化方向

### 1. 输出控制
- 采用固定回答骨架：`Conclusion / Evidence / NextStep`。
- 统一输出后处理：去寒暄、去重复、压缩空白、限制长度。
- 按场景限制 `max_tokens`（问答、工具、代码问题分别设置）。

### 2. 上下文压缩
- 由“只截断历史”升级为“最近窗口 + 会话摘要”双层记忆。
- 只保留工具结果中的关键字段，不把整段冗长结果回灌给模型。
- 对重复历史问题做去重，避免重复注入。

### 3. Wiki 检索最小化
- 禁止全量 wiki 注入，改为 `top-k` 片段检索。
- 用关键词命中 + 调用关系权重排序，提高相关性。
- 摘要动态长度：关键符号允许稍长，普通符号强制短摘要。

### 4. 工具调用收敛
- 统一工具回填 JSON 结构，避免自然语言歧义。
- 对错误做类型化（参数错误、工具不存在等），限制可重试次数。
- `write_file` 统一走 v2 内核，主协议为 `edits(JSON)`（兼容 compact `op/s/e/t`）。

### 5. 可观测性与 A/B 验证
- 记录每次交互的关键指标：工具轮次、工具调用数、回复长度。
- 采用同一问题集做 A/B 对照，避免样本漂移导致的误判。

- `llmServer/llmPolling.py`：

  - 新增会话摘要压缩；

  - 增加指标落盘（jsonl）。
- `llmServer/CreateCodeNode.py`：
  - 保持 index_only 轻量输出；
  - 摘要请求仅传片段；
  - 引入动态摘要字数预算。
- `RadishTools/src/FileExecutor/core/WriteFile.py`：
  - 历史实现保留用于兼容参考；
  - llm tools 默认链路已切换到 v2 写入服务。

## A/B 验证建议
- 问题集：20-50 条固定样本（问答/工具/代码三类均衡）。
- 对比组：
  - A：旧策略；
  - B：当前最小输出策略。
- 关注指标：
  - 平均输出长度（字符或 token）；
  - 平均工具调用次数；
  - 首答成功率；
  - 平均响应耗时。

## 风险与处理
- 输出过短导致信息不足：保留最小三段结构，必要时自动扩一条 `Evidence`。
- 检索裁剪过头：当置信度不足时自动扩大 `top-k`。
- 协议升级兼容问题：新协议优先，异常时回退 legacy。
